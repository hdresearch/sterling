### Timing Layer

A tracing subscriber layer which adds performance monitoring and operation tracking and optional return value capture. 

Add the macro `#[instrument(ret, err, skip_all)` to capture the return value, error, and ignore all parameters passed into the function.
Update: Most functions that can support `ret` or `err` have them on their instrument. Ones that cant support `ret` need to implement debug.
- Currently the macros arent capturing `ret` or `err` so need to add that on the certain macros as needed.

## Usage
In your main process/function you need to initialize the tracing layer with a writer that implements the `TimingWriter` trait then subscribe to it with 

```
let subscriber = tracing_subscriber::registry()
        .with(timing)
        .with(any additional layers you want)
```

### Timing Writer
You must provide a writer within the main crate to initialize the timing layer with. 

It must implement timing writer to handle how the records are stored and logged. An example writer is provided that uses a hashmap to store partial records and finalizes the record to a timing.log file.

## How it works

### Layer Implementation
The `TimingLayer` implements the `tracing_subscriber::Layer` trait with these methods:

#### `on_new_span()`
- Extracts `operation_id` from span attributes using `OperationIdVisitor`
- If no operation_id is found, inherits from parent span or generates a new UUID
-  ^Updt: Op_id is now optional and isnt generated by our layer
- Creates a new `TimingEvent` and stores it via the `TimingWriter`
- Stores the operation_id in span extensions for child spans to inherit

#### `on_enter()` 
- Records the start time (`TimingStartInstant`) when a span is entered
- Uses span extensions to store timing data without key collisions

#### `on_event()`
- Captures return values and error information from instrumented functions
- Uses `RetErrVisitor` to extract `return` and `error` fields from events
- Updates the existing timing event with return value data
- Distinguishes between successful returns and `Result::Err` cases

#### `on_close()`
- Calculates elapsed time from the stored start time
- Builds the full hierarchical span name by walking up parent spans
- Updates the timing event with final timing and naming data
- Calls `finalize_record()` to persist the completed timing data

### Visitor Pattern
The layer uses two visitors to extract data from tracing spans and events:

#### `OperationIdVisitor`
- Looks for a string field named `"operation_id"` in span attributes
- Called when `attrs.record(&mut visitor)` processes span creation

#### `RetErrVisitor` 
- Extracts return values from instrumented function events
- Looks for `"return"` field (successful results) and `"error"` field (errors)
- Sets `is_error` flag when the return value was a `Result::Err`
- Called when `event.record(&mut visitor)` processes function returns

### Data Flow
1. Function with `#[instrument]` starts -> `on_new_span()` creates timing event
2. Function begins execution -> `on_enter()` records start time
3. Function returns -> `on_event()` captures return value
4. Function completes -> `on_close()` calculates timing and persists data
